EXECUTIVE SUMMARY
1.1 Project Overview
Budget Mate is a cross-platform personal finance tracking system that helps users manage expenses and budgets through a web app and Android app connected to one backend service.
The system consists of:
• A backend API built with Spring Boot
• A web application built with React
• A native mobile application for Android (Kotlin with XML-based UI)
• A relational database using PostgreSQL via Supabase
The clients communicate securely with the backend via REST APIs. The system implements JWT authentication, Google OAuth login, SMTP-based email flows, exchange-rate API integration, profile image upload, and realtime dashboard auto-updates (WebSocket/STOMP support on backend, web, and Android clients).

1.2 Objectives
1. Develop a secure expense and budget tracking platform for web and Android users.
2. Implement JWT-based authentication and Google OAuth login.
3. Apply Role-Based Access Control (Admin and User roles).
4. Implement full CRUD operations for expense and budget management.
5. Integrate a real external API for exchange-rate retrieval and currency display support.
6. Send operational emails using SMTP (welcome + password reset).
7. Follow a layered backend architecture with proper separation of concerns.

1.3 Scope
Included Features:
• User registration and login
• JWT authentication
• Google OAuth login
• Role-based access control
• Expense management (CRUD)
• Budget management (CRUD)
• External API integration (exchange rates)
• Email notifications (welcome + password reset)
• Profile management (name/password/image)
• Realtime dashboard updates (backend publish + web/mobile subscribe)
• DTO usage (no password exposure)
• Global exception handling
• RESTful API design
Excluded Features:
• iOS version of the application
• AI-based financial forecasting
• Investment portfolio tracking
• Payment gateway features
• PDF export

2.1 Project Overview

2. Administrators

Solution:
Budget Mate provides a secure platform that allows users to:
• Record expenses
• Create and monitor category budgets
• View spending summaries
• Display the selected currency indicator for amounts

2.2 Core User Journeys
Journey 1: User Registration and Setup
1. User opens web or Android app
2. Registers using email/password OR logs in with Google OAuth
3. Password is hashed using BCrypt (for local accounts)
4. JWT token is issued
5. Welcome email is sent after registration
6. User creates initial budget and starts tracking expenses

Journey 2: Expense and Budget Tracking
1. User logs in
2. Creates a new expense entry
3. Expense is saved in database and linked to user
4. If linked to a budget, budget spent amount is recalculated
5. Dashboard reflects updated financial data
6. User can independently switch Total Budget and Total Spent cards between current month and current year views

Journey 3: Admin Monitoring
1. Admin logs into the system
2. Opens admin dashboard
3. Views user list and status info
4. Performs allowed administrative actions (view users, delete eligible non-admin accounts)

2.4 Detailed Feature Specifications
Feature: Authentication and Security
• Screens: Registration, Login, Forgot Password, Reset Password, Profile
• Fields: Name, Email, Password, Current Password, New Password
• Validation: email format, unique email, required fields, password constraints
• API Endpoints: POST /api/v1/auth/register, POST /api/v1/auth/login, POST /api/v1/auth/logout, GET /api/v1/auth/me, POST /api/v1/auth/forgot-password, POST /api/v1/auth/reset-password
• Security: JWT-based auth, BCrypt password hashing, protected routes, encrypted token storage in web and Android clients

Feature: Role-Based Access Control
• Roles: USER, ADMIN
• Access Rules: secured endpoints enforced by Spring Security + method-level authorization
• Admin Restrictions: admin-only user listing and user deletion endpoints; admin self-delete is blocked
• Client Restriction: UI-level admin feature visibility based on role
• API Endpoints: protected endpoints under /api/v1/users and other modules

Feature: Expense Management
• Screens: Add Expense, Edit Expense, Expense List, Expense Details
• Fields: title, description, amount, currency, category, expenseDate, isRecurring, budgetId(optional)
• Validation: amount > 0, required fields cannot be empty, valid date
• Functions: create expense, list user expenses, update expense, delete expense
• Persistence: stored in PostgreSQL (Supabase-hosted) expenses table linked to authenticated user
• API Endpoints:
  o POST /api/v1/expenses
  o GET /api/v1/expenses
  o GET /api/v1/expenses/{id}
  o PUT /api/v1/expenses/{id}
  o DELETE /api/v1/expenses/{id}

Feature: Budget Management
• Screens: Create Budget, Budget Overview Dashboard
• Fields: category, limitAmount, currency, startDate(optional), endDate(optional), notes(optional)
• Functions: create budget, compute spent/remaining amounts, update/delete budget, validate scheduled date ranges, show linked expenses in a structured list inside budget details, show recency-based category sequence indicators for duplicate categories (1 newest, 2 next, etc.), view linked-expense budget details without exposing numeric budget IDs in the popup, and delete linked expenses when a budget is deleted
• Persistence: stored in PostgreSQL (Supabase-hosted) budgets table linked to authenticated user
• API Endpoints:
  o POST /api/v1/budgets
  o GET /api/v1/budgets
  o GET /api/v1/budgets/{id}
  o PUT /api/v1/budgets/{id}
  o DELETE /api/v1/budgets/{id}

Feature: External API Integration (Currency Exchange)
• Screens: Dashboard currency display
• Data Displayed: selected currency indicator and exchange-rate data
• Purpose: support currency indicator display using available exchange-rate data
• Integration: ExchangeRate-API consumption with fallback behavior
• API Endpoints:
  o GET /api/v1/exchange-rates
• Requirement: exchange-rate data retrieval must fail safely when external API is unavailable (fallback response)

Feature: Email Sending (SMTP Notifications)
• Trigger Events: user registration, forgot-password workflow
• Email Types: Welcome Email, Password Reset Email
• Configuration: SMTP server configuration required
• Security: no sensitive credentials sent in email body
• API Endpoints (triggered by auth flows):
  o POST /api/v1/auth/register
  o POST /api/v1/auth/forgot-password

Feature: Profile Management and File Upload
• Screens: Profile settings (web and Android)
• Fields: display name, current/new password, profile image file
• Validation: image type and max file size validation (5MB)
• Functions: update name, change password, upload/download profile image
• API Endpoints:
  o GET /api/v1/users/me
  o PUT /api/v1/users/me/name
  o PUT /api/v1/users/me/password
  o POST /api/v1/users/me/profile-image
  o GET /api/v1/users/me/profile-image

2.5 Acceptance Criteria
AC-1: User Registration
Given I am a new user
When I register with valid credentials
Then my password is hashed
And I receive a welcome email
And I receive a JWT token

AC-2: User Login
Given I am a registered user
When I log in with valid credentials
Then my credentials are validated
And a JWT token is issued
And I can access protected endpoints

AC-3: Google OAuth Login
Given I choose to log in using Google
When authentication is successful
Then my account is created or linked
And I receive a JWT token
And I can access the application

AC-4: Role-Based Access Control
Given I am logged in as a USER
When I attempt to access an admin-only endpoint
Then access is denied
Given I am logged in as an ADMIN
When I access admin endpoints
Then access is granted

AC-5: Expense Creation
Given I am logged in
When I create an expense with valid data
Then the expense is stored in the database
And it is linked to my user account
And linked budget totals are updated when applicable

AC-6: Expense Update
Given I have an existing expense
When I update the expense with valid data
Then the changes are saved
And related budget totals are recalculated when needed

AC-7: Expense Deletion
Given I have an existing expense
When I delete the expense
Then the expense is removed from the database
And linked budget totals are recalculated when needed

AC-8: Budget Creation
Given I am logged in
When I create a budget with valid data
Then the budget is stored in the database
And it is linked to my user account

AC-9: Budget Monitoring
Given I have a budget
When expenses are recorded
Then spent and remaining values are computed correctly
And users can see budget progress in the dashboard/UI
And users can switch dashboard total cards between Month and Year summaries
And when a budget is deleted, all expenses linked to that budget are also deleted

AC-10: Budget Deletion
Given I have an existing budget
When I delete that budget
Then the budget is removed from the database
And all expenses linked to that budget are also deleted
And dashboard data is refreshed to reflect the deletion

AC-11: External API Integration (Exchange Rates)
Given I view currency data
When the system fetches exchange rates
Then exchange data is returned
And selected currency display remains available for users

AC-12: Email Notification
Given I register or request password reset
When the flow completes
Then an email is sent through SMTP workflow

3.0 NON-FUNCTIONAL REQUIREMENTS
3.1 Performance Requirements
• API response time target: <= 2 seconds for common operations under normal load
• Authentication flow should remain responsive for login/register
• Exchange rate retrieval should fail fast and return fallback when external service is unavailable
• Dashboard refresh operations should complete promptly for user interaction
• Database queries should be optimized through repository paging and targeted filtering

3.2 Security Requirements
• Passwords hashed with BCrypt before persistence
• JWT signed and validated for protected endpoints
• Revoked token tracking is enforced on logout
• Role-based endpoint protection enforced at backend level
• Sensitive data (passwords/tokens) never exposed in API responses
• Input validation applied on request DTOs
• JWT stored securely on Android (EncryptedSharedPreferences)
• JWT stored with client-side encryption utility on web

3.3 Reliability and Availability Requirements
• Global exception handling provides consistent error responses
• External exchange API failure does not crash core app functionality
• WebSocket failures should not block core expense/budget workflows
• Email sending failures should not corrupt primary transaction data

3.4 Scalability Requirements
• Backend architecture supports modular service growth
• Database schema supports increasing user and transaction data
• Pagination on list endpoints supports larger datasets

3.5 Usability Requirements
• Android and web flows should remain intuitive and task-focused
• Validation and error messages must be clear to users
• Budget progress and spending values should be readable and understandable

3.6 Maintainability Requirements
• Backend follows layered architecture (Controller, Service, Repository)
• DTO pattern separates entities from API responses
• Code follows organized module boundaries for auth, users, expenses, budgets, integrations
• Logging is present for operational debugging

3.7 Compatibility Requirements
• Mobile app supports Android API 24 and above
• Backend interoperates with web app, Android app, and REST clients
• Local web client origin is configured for CORS

3.8 Data Integrity Requirements
• Every expense and budget record is tied to an authenticated user
• Ownership checks prevent unauthorized resource access
• Budget and expense recalculation logic remains consistent after create/update/delete operations
• Deleting a budget also removes its linked expense records for the same user
• User emails are unique

Technology Stack:
• Backend: Java 17, Spring Boot, Spring Security (JWT + RBAC), Spring Data JPA, WebSocket/STOMP
• Database: PostgreSQL via Supabase
• Web: React
• Mobile: Kotlin, XML layouts (Activity + Fragments), Retrofit
• Build Tools: Maven (Backend), Gradle (Android)
• Deployment: Standard Java hosting + web static hosting + Android APK

6.0 DATABASE DESIGN
6.1 Entity Relationship Diagram

Detailed Relationships:
• One-to-Many: User -> Expenses (one user can own multiple expenses)
• One-to-Many: User -> Budgets (one user can own multiple budgets)
• One-to-Many: User -> UserRoles (a user may hold one or more roles)
• Many-to-One: Expense -> User (each expense belongs to one user)
• Many-to-One: Budget -> User (each budget belongs to one user)
• Many-to-One: Expense -> Budget (optional budget link per expense)

Key Tables:
1. users - User accounts and profile/auth metadata
2. user_roles - User role mapping (USER/ADMIN)
3. expenses - User expense records
4. budgets - User budget records
5. revoked_tokens - Revoked JWT records for logout handling

Table Structure Summary:
• users: id, name, email, password, provider, provider_id, image_url, enabled, email_verified, last_login_at, created_at, updated_at
• user_roles: user_id, role
• expenses: id, user_id, budget_id, title, description, amount, currency, category, expense_date, is_recurring, created_at, updated_at
• budgets: id, user_id, category, limit_amount, spent_amount, currency, start_date, end_date, notes, created_at, updated_at
• revoked_tokens: id, token, expires_at, created_at
